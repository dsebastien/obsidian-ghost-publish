# 2026-06-17

## Feature — Markdown footnote support (issue #7 / GH #1)

GitHub issue #1: "Support for Markdown footnotes." Footnotes are supported by Ghost, but the plugin's HTML pipeline lost them: stock `marked` has no footnote handling, so `text[^1]` / `[^1]: …` were mis-parsed as broken reference-style links (e.g. `[^1]` → `<a href="note">^1</a>`).

### What was done

- New `src/app/services/footnotes.ts` — `parseMarkdownWithFootnotes(markdown)`: a `marked` extension set + `postprocess` hook that renders:
    - references `[^id]` → `<sup class="footnote-ref"><a href="#fn-N" id="fnref-N">N</a></sup>`,
    - inline footnotes `^[…]` (bracket-balanced so nested `[...]` survives),
    - definitions `[^id]: …` (incl. multi-line indented continuations) hoisted into a trailing `<section class="footnotes" data-footnotes><ol>…</ol></section>` with back-references.
- `build-html.ts` now calls `parseMarkdownWithFootnotes` instead of `marked.parse`.
- `footnotes.spec.ts` — 9 cases (numbering by reference order, re-reference id dedupe, dropped-unreferenced, dangling-as-literal, inline footnotes, markdown inside defs, multi-line continuations).

### Key decisions

- **Custom extension, no new dependency.** `marked-footnote` exists but the project rule is "avoid large dependencies"; the extension is small and self-contained.
- **Numbering by first-reference order**, matching markdown-it-footnote. Definitions never referenced are dropped; references with no definition stay literal text. Safe because all block (def) tokens are lexed before any inline (ref) token, so the def map is complete when a ref tokenizer checks it.
- **Anchors keyed by footnote number** (`fn-N` / `fnref-N`), not the raw id text — avoids HTML-injection / invalid-id issues from arbitrary Obsidian ids. Re-references reuse the number but only the first emits `id="fnref-N"` to keep element ids unique.
- **Per-call `Marked` instance** with closure state, so numbering never leaks across notes.
- Output is standard footnote HTML; Ghost's `source=html` → lexical conversion preserves the inline `<sup>` anchors and the footnotes `<section>` (as an HTML card). Documented as BR-PROC-5.
