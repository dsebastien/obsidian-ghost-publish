# 2026-06-03

## Fix — settings tab crash on Obsidian 1.13+ (`[Immer] produce` error)

Plugin failed to load on the new Obsidian settings window with:

```
Plugin failure: ghost-publish Error: [Immer] The first or second argument to `produce` must be a function
    at GhostPublishSettingTab.update (...)
    at addSettingTab (app.js)
    at GhostPublishPlugin.onload (...)
```

Renamed `GhostPublishSettingTab.update(mutator)` → `mutateSettings(mutator)` in `src/app/settings/settings-tab.ts` (definition + all 17 call sites).

## Root cause

Obsidian 1.13.0 introduced a declarative settings API and added a reserved method to the `SettingTab` base class:

```ts
// obsidian.d.ts, @since 1.13.0
update(): void;   // "Called by addSettingTab() ..."
```

`addSettingTab()` now calls `tab.update()` with **no arguments** at registration time. Our private helper was also named `update(mutator)`, so it **shadowed** Obsidian's method. Obsidian invoked it with no args → `mutator === undefined` → `produce(this.plugin.settings, undefined)` → Immer `die(6)`.

It only surfaced now because:

- The method existed and worked on older Obsidian (nothing called `tab.update()`).
- The pinned `obsidian` types were **1.10.3**, which predate `SettingTab.update()`, so `tsc` never flagged the accidental override. The user's running app is 1.13.0+.

## Takeaway / business rule

Method names on a `PluginSettingTab` subclass must not collide with Obsidian's `SettingTab` base members. Reserved (1.13.0): `display`, `hide`, `update`, `getSettingDefinitions`, `getControlValue`, `setControlValue`, `settingItems`, `icon`, `containerEl`, `app`. Verified current class only intentionally defines `display`; all helpers use plugin-specific names (`mutateSettings`, `render*`, `summarisePreset`).

## Follow-ups (preventative)

- Bumped the `obsidian` dev types from the stale 1.10.3 (which predated `SettingTab.update()`) to **1.13.0**. With current types, `noImplicitOverride` now flags any accidental override of a `SettingTab`/`Plugin` base member at compile time.
- The bump surfaced two latent `override`-modifier errors the old types had hidden, now fixed:
    - `src/app/plugin.ts` — `override settings` (Obsidian 1.13.0 `Plugin` declares `settings?: unknown`).
    - `src/app/settings/settings-tab.ts` — `override display()`.
- Added a reserved-`SettingTab`-member rule to `AGENTS.md` (Commands & settings), mirrored into the upstream `obsidian-plugin-template` (which got the same types bump + `override` fixes).
